Einführung in PLS

Wie ein Katastrophenschutzsystem Patienten triagiert, Rettungskräfte koordiniert und Leben rettet

Wenn eine Katastrophe eintritt – ein Zugunfall, ein Hauseinsturz oder ein Massenkarambolage – zählt jede Sekunde. PLS (Patientenleitsystem) ist das Nervensystem der Notfallreaktion. Es hilft Sanitätern und Rettungskräften, Patienten schnell zu bewerten und zu kategorisieren, ihre Standorte zu verfolgen und die Versorgung über einen Einsatzort hinweg zu koordinieren.

Dieser Kurs führt dich durch den Code, der dieses System antreibt. Du wirst sehen, wie ein einfacher QR-Code-Scan eine Kaskade von Entscheidungen auslöst, wie Patientendaten von der Front in einen zentralen Situationsraum fließen, und wie Farben eine Geschichte über die dringendsten Fälle erzählen.

Ein Tag im Leben einer Sanitäterin

Folgen wir Maya, einer Rettungssanitäterin, die auf einen Großeinsatz reagiert. Hier ist, was unter der Haube passiert, wenn sie PLS nutzt.

1
Ankunft
Maya kommt am Einsatzort an und scannt ihren QR-Code um sich anzumelden. Das System überprüft ihre Anmeldedaten.
↓
2
Einsatzort-Setup
Das System fragt: Welcher Einsatzort ist das? Maya wählt oder erstellt den Katastrophenort aus. Dies gruppiert alle Patienten und Rettungskräfte zusammen.
↓
3
Patient finden
Maya findet eine verletzte Person und scannt deren QR-Code. Das System registriert diesen Patienten in der Szene und öffnet die Triagebewertung.
↓
4
Bewerten & Kategorisieren
Maya beantwortet drei kritische Fragen hintereinander (TriageSeite1, 2, 3). Basierend auf ihren Antworten weist das System eine Farbe zu: 🟢 grün (gehfähig), 🔴 rot (dringend), 🟡 gelb (später), ⚫ schwarz (verstorben).
↓
5
Verletzungen markieren
Maya kann eine Körperkarte berühren, um zu markieren, wo sich Verletzungen befinden. Dieser visuelle Datensatz hilft anderen Sanitätern zu wissen, was sie erwartet.
↓
6
Situationsraum sieht alles
Inzwischen sieht ein Koordinator in einem Kommandoposten eine Live-Karte mit allen Patienten, farblich codiert nach Dringlichkeit. Er kann Krankenwagen und Ressourcen entsprechend einsetzen.
💡 Aha! — Informationsfluss

Das, was dieses System leistungsfähig macht, ist nicht, was ein Sanitäter sieht – es ist, dass seine Entscheidungen (Triagefarbe, Standort, Verletzungsmarker) sofort den Koordinator erreichen, der Ressourcen zuweist. Das System ist eine Brücke zwischen dem Chaos der Szene und der Ordnung des Kommandopostens.

Der Entscheidungsbaum

PLS verwendet ein einfaches, aber leistungsstarkes dreiteiliges Triageprotokoll. Schauen wir uns an, wie der Code diese Entscheidungen trifft.

🚶

Seite 1: Mobilität

F: Kann der Patient gehen?

Wenn JA → 🟢 GRÜN (gehfähig, niedrigste Priorität)

Wenn NEIN → weiter zu Seite 2

Sonderfall: Todeszeichen → ⚫ SCHWARZ

💨

Seite 2: Atmung

F: Atmet der Patient angemessen?

Wenn NEIN → 🔴 ROT (Bedrohung für das Leben, höchste Priorität)

Wenn JA → weiter zu Seite 3

🩹

Seite 3: Blutung

F: Gibt es schwere unkontrollierte Blutungen?

Wenn JA → 🔴 ROT (Priorität wie Atmung)

Wenn NEIN → 🟡 GELB (verzögerte Priorität)

💡 Aha! — Warum drei Seiten?

Jede Seite stellt eine binäre (Ja/Nein) Frage, die Mehrdeutigkeit ausschließt. Ein Sanitäter unter Stress muss keine komplexen Urteile treffen – nur beobachten und antworten. Der Code macht den Rest. Das ist klinische Einfachheit.

Wie Routes funktionieren

Hier ist der Router, der URLs Seiten zuordnet:

export const router = createBrowserRouter([ { path: '/qrAuthenticator', element: <QRAuthenticator /> }, { path: '/TriagePage1', element: <ProtectedRoute element={<TriagePage1 />} /> }, { path: '/SituationRoomTable', element: <ProtectedRoute element={<SituationRoomTable />} /> }, // ... more routes ]);
createBrowserRouter — Erstellt ein Navigationssystem
path — Die URL (z.B. /TriagePage1)
element — Welche Seite soll angezeigt werden
ProtectedRoute — Umhüllt Seiten, die ein gültiges Login-Token benötigen
Wenn der Benutzer nicht angemeldet ist, werden sie stattdessen zur Anmeldung umgeleitet
💡 Aha! — Geschützte Routes

ProtectedRoute ist eine Umhüllung, die sagt: „Nur Benutzer mit einem gültigen Token können diese Seite sehen." Dies verhindert, dass jemand ohne Anmeldung zufällig auf die Triageseiten zugreift.

Wie ein API-Endpunkt funktioniert

Wenn Maya die Triagefarbe eines Patienten speichert, passiert folgendes:

[HttpPost("api/persons")] public async Task Index( [FromBody] PersonsRequest req) { var patients = await _context .Patients .Where(p => p.OperationSceneId == req.operationSceneId) .ToListAsync(); return Ok(result); }
[HttpPost(...)] — Diese Funktion hört auf HTTP POST-Anfragen
[FromBody] — Die Anfragedaten (Einsatzort-ID) kommen vom Anfragekörper
_context.Patients — Zugriff auf die Patienttabelle der Datenbank
.Where(...) — Filter: Nur Patienten auf diesem Einsatzort
.ToListAsync() — Abfrage ausführen (async = nicht blockierend)
return Ok(result) — Patientenliste an das Frontend zurückschicken
💡 Aha! — Request/Response

Das Frontend sendet eine Anfrage (z.B. „Alle Patienten in Einsatzort 42 abrufen"). Das Backend findet die Daten und sendet eine Antwort (eine JSON-Liste von Patienten) zurück. Dieses Hin und Her ist, wie die beiden Hälften sprechen.

Vom Klick zur Datenbank

Verfolgen wir, was passiert, wenn Maya eine Triagefarbe zuweist. Jeder Schritt ist choreographiert.

1
Benutzer klickt auf „Rot"
Maya klickt auf die Schaltfläche, die den Patienten als dringend markiert.
↓
2
React verarbeitet den Klick
const handleRed = () => { setTriageColor(TRIAGE_COLOR.ROT); }
↓
3
API-Aufruf gesendet
Das Frontend sendet eine HTTP POST zum Backend mit der Patienten-ID und der Farbe „rot"
↓
4
Backend empfängt & validiert
Der C#-Controller prüft: Ist die Anfrage autorisiert? Ist die Patienten-ID gültig?
↓
5
Datenbankupdate

↓
6
Bestätigung zurückgesendet
Backend gibt zurück: „Erfolg! Patient 42 ist jetzt rot."
↓
7
Frontend aktualisiert
React aktualisiert den lokalen Status und wechselt zur nächsten Seite.
↓
8
Situationsraum sieht Live-Update
Der Bildschirm des Koordinators aktualisiert sich und zeigt den Patienten mit einem roten Marker auf der Karte an.
💡 Aha! — Der vollständige Stack

Ein Schaltflächenklick schafft eine Reise: Frontend → Netzwerk → Backend → Datenbank → Netzwerk → Frontend → Live-Kartenerneuerung. Dies ist der Grund, warum das System „Full Stack" genannt wird – es berührt jede Schicht. Ein Bug in einer beliebigen Schicht bricht den gesamten Fluss.

Der Situationsraum: Zentrale Koordination

Während Sanitäter im Einsatz arbeiten, sitzt ein Koordinator an einem Kommandoposten mit einer Karte, auf der jeder Patient und sein Status angezeigt wird.

🗺️

Live-Karte

Leaflet-Karte mit Patientenstandorten als farbcodierte Marker. Ein Blick zeigt, wo sich die dringenden Fälle befinden.

📊

Zusammenfassungsabzeichen

Anzahl der Patienten nach Farbe (z.B. „5 rot, 12 gelb, 8 grün"). Hilft dem Koordinator, Krankenwagen zu verteilen.

🔄

Echtzeit-Aktualisierung

Während Sanitäter Triagefarben zuweisen, wird der Situationsraum automatisch aktualisiert. Keine manuelle Aktualisierung erforderlich.

Wie SituationRoomTable Farben anzeigt

Die Komponente zählt Patienten nach Triagefarbe und zeigt Zusammenfassungsabzeichen an:

const getTriageDisplay = (color) => { const normalized = color?.trim().toLowerCase(); if (!normalized) { return { label: 'Keine Farbe', backgroundColor: 'white' }; } if (isTriageColor(normalized)) { return { backgroundColor: TRIAGE_COLOR_BACKGROUND [normalized] }; } };
getTriageDisplay() — Funktion, die eine Farbe nimmt und entscheidet, wie sie angezeigt wird
.trim().toLowerCase() — Eingabe bereinigen (Leerzeichen entfernen, zu Kleinbuchstaben konvertieren)
if (!normalized) — Wenn leer/null, zeige „keine Farbe zugewiesen"
isTriageColor() — Validierung: Ist dies eine der fünf offiziellen Farben?
TRIAGE_COLOR_BACKGROUND — Schau den Hex-Farbcode nach (z.B. 'rot' → '#c85a54')
Gibt ein Objekt mit Anzeigeinformationen zurück (Farbe, Text, Rand)
5 🟢 Grün
12 🔴 Rot
18 🟡 Gelb
3 🔵 Blau
2 ⚫ Schwarz
💡 Aha! — Verteilte Verantwortung

Die Schönheit dieses Designs: Sanitäter konzentrieren sich auf Bewertung (welche Farbe hat dieser Patient?). Der Koordinator konzentriert sich auf Zuweisung (schicke Krankenwagen zuerst zu den Roten). Jede Person hat einen Job, und das System verbindet sie.

Wichtigste Erkenntnisse

🎯

PLS ist ein System, kein nur Code

Es ist eine Konversation zwischen Sanitätern, die Patienten bewerten, und Koordinatoren, die Ressourcen zuweisen. Der Code ist nur das Fahrzeug für diese Konversation.

🔄

Daten fließen bidirektional

Sanitäter schicken Entscheidungen (Triagefarben, Verletzungsstandorte) zur Datenbank. Koordinatoren ziehen Live-Daten zum Situationsraum herunter. Das Backend ist der Hub.

⚡

Geschwindigkeit zählt

Eine 2-Sekunden-Verzögerung bei der Aktualisierung des Situationsraums könnte bedeuten, dass ein Krankenwagen an den falschen Ort geschickt wird. Echtzeit-Synchronisation ist kein nice-to-have – sie ist kritisch.

🛡️

Sicherheit ist medizinisch

QR-Code-Authentifizierung, geschützte Routes und Datenbankberechtigungen sind kein behördlicher Overhead – sie verhindern, dass ungeschulte Menschen auf medizinische Daten zugreifen oder kritische Aktionen auslösen.

🎓 Abschließender Gedanke

Dieser Kurs zeigte dir, wie du Code wie eine Geschichte liest. Jede Funktion, jede Komponente und jede Datenbanktabelle ist ein Charakter mit einer Rolle. Wenn du einen Klick durch das gesamte System verfolgen kannst – von React zu C# zu MySQL und zurück – verstehst du nicht nur, was der Code tut, sondern warum jedes Stück existiert. Das ist die Kraft des Verständnisses der Architektur.
